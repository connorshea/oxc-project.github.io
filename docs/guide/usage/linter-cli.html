Command-line Interface â€‹

Usage â€‹

oxlint [-c=<./eslintrc.json>] [--fix] [PATH]...

Omitting PATH defaults to linting the current working directory.

Useful Options and Examples â€‹

  • --deny-warnings turn warnings into errors, useful for failing CI with exit code 1.
  • -D all deny (turn on) all rules.
  • -A all -D eqeqeq run a single rule.
  • -D correctness -D perf -A debugger -A eqeqeq deny (turn on) correctness and perf rules and allow (turn off) the debugger and eqeqeq rule.
  • -c ./eslintrc.json use the rules field to configure the rules, as documented in ESLint. Only json format is supported.

Basic Configuration â€‹

  • -c, --config=<./eslintrc.json> â€” ESLint configuration file (experimental)

    • only .json extension is supported

  • --tsconfig=<./tsconfig.json> â€” TypeScript tsconfig.json path for reading path alias and project references for import plugin

Allowing / Denying Multiple Lints â€‹

For example -D correctness -A no-debugger or -A all -D no-debugger.

The default category is -D correctness.

See rules for rule names.

The categories are:

  • correctness - code that is outright wrong or useless
  • suspicious - code that is most likely wrong or useless
  • pedantic - lints which are rather strict or have occasional false positives
  • style - code that should be written in a more idiomatic way
  • nursery - new lints that are still under development
  • restriction - lints which prevent the use of language and library features
  • all - all the categories listed above
  • -A, --allow=NAME â€” Allow the rule or category (suppress the lint)
  • -D, --deny=NAME â€” Deny the rule or category (emit an error)

Enable Plugins â€‹

  • --import-plugin â€” Enable the experimental import plugin and detect ESM problems. It is recommended to use along side with the --tsconfig option.
  • --jsdoc-plugin â€” Enable the experimental JSDoc plugin and detect JSDoc problems
  • --jest-plugin â€” Enable the Jest plugin and detect test problems
  • --jsx-a11y-plugin â€” Enable the JSX-a11y plugin and detect accessibility problems
  • --nextjs-plugin â€” Enable the Next.js plugin and detect Next.js problems
  • --react-perf-plugin â€” Enable the React performance plugin and detect rendering performance problems

Fix Problems â€‹

  • --fix â€” Fix as many issues as possible. Only unfixed issues are reported in the output

Ignore Files â€‹

  • --ignore-path=PATH â€” Specify the file to use as your .eslintignore

  • --ignore-pattern=PAT â€” Specify patterns of files to ignore (in addition to those in .eslintignore)

    The supported syntax is the same as for .eslintignore and .gitignore files You should quote your patterns in order to avoid shell interpretation of glob patterns

  • --no-ignore â€” Disables excluding of files from .eslintignore files, --ignore-path flags and --ignore-pattern flags

  • --symlinks â€” Follow symbolic links. Oxlint ignores symbolic links by default.

Handle Warnings â€‹

  • --quiet â€” Disable reporting on warnings, only errors are reported
  • --deny-warnings â€” Ensure warnings produce a non-zero exit code
  • --max-warnings=INT â€” Specify a warning threshold, which can be used to force exit with an error status if there are too many warning-level rule violations in your project

Output â€‹

  • -f, --format=ARG â€” Use a specific output format (default, json, unix, checkstyle, GitHub)

Miscellaneous â€‹

  • --threads=INT â€” Number of threads to use. Set to 1 for using only 1 CPU core

Available positional items: â€‹

  • PATH â€” Single file, single path or list of paths

Available options: â€‹

  • --rules â€” list all the rules that are currently registered
  • -h, --help â€” Prints help information
